Rust 内存空间布局

一个 Rust 进程启动后，操作系统分给它的虚拟地址空间主要分为 5 块。下面按「高地址 → 低地址」排列（和大多数平台的实际方向一致）。

栈Stack
函数调用帧、局部变量、参数、返回地址。自动分配/释放，速度极快，大小在编译期已知。向低地址增长 ↓
i32 / f64 / bool struct（无堆字段时） String/Vec 的「头部」(ptr,len,cap) &T 引用 Box<T> 的指针
堆Heap
运行时动态分配，大小可变。通过分配器（Rust 默认用系统 allocator）申请，由所有者负责释放（RAII / drop）。向高地址增长 ↑
String 的字节缓冲 Vec<T> 的元素 Box<T> 指向的数据 Rc / Arc 的计数+数据
String 在栈上只是 { ptr, len, cap } 三个字段；
ptr 这根箭头 → 指向这里真正的字符数据。
已初始化数据段.data / .rodata
编译期就确定初值的全局/静态数据。.rodata 只读（字符串字面量、不可变 static），.data 可读写（可变 static）。
字符串字面量 "hello" static 变量 const（内联，通常不占独立位置）
未初始化数据段.bss
初值为 0 或未初始化的全局/静态数据。二进制里不存实际内容，只记录大小，加载时清零，省空间。
static mut 未初始化 零初始化的全局数组
代码段.text
编译后的机器指令。只读、可共享（同一程序多个进程共用一份）。

核心心智模型：栈上放「已知大小、随函数生死」的数据；堆上放「运行时才知道大小、需要拥有者管理」的数据。 String / Vec / Box 都是「栈上一个指针头 + 堆上一块数据」的组合，箭头从栈指向堆。

一段代码，数据各去哪了

static GREETING: &str = "hi";   // ① 字面量 → .rodata

fn main() {
    let n: i32 = 42;            // ② 值 → 栈
    let s = String::from("hello"); // ③ 头在栈，字节在堆
    let b = Box::new(7);         // ④ 指针在栈，7 在堆
    println("{GREETING}");       // ⑤ 指令 → .text
}
代码数据位置
① "hi".rodata 只读段
② n = 42栈
③ String 头部栈 (ptr,len,cap)
③ String 字节堆 ("hello")
④ Box 指针栈
④ 整数 7堆
⑤ 函数指令.text 代码段

常见类型放哪、多大

类型 / 写法数据位置栈上大小说明
i32 / f64 / bool / char栈4 / 8 / 1 / 4 字节纯值，编译期已知大小
[T; N] 定长数组栈N × size_of(T)整体在栈上
&T / &mut T栈8（胖指针 16）只是个指针，数据在别处
&str（字面量）栈指针 → .rodata16（胖指针）指向只读段，'static
String栈头部 → 堆缓冲24ptr+len+cap，数据在堆
Vec<T>栈头部 → 堆元素24同 String 结构
Box<T>栈指针 → 堆8独占堆内存
static 变量.data / .rodata—固定地址，全程存在
const无固定位置—编译期内联，使用点复制
和 JS 对照：JS 几乎所有「对象/字符串/数组」都在堆上，变量只是引用；基本类型（number/bool）按值传递。 Rust 默认把已知大小的值直接放栈上（包括 struct 和数组），只有 String/Vec/Box 这类「拥有型容器」才把数据放堆上。 这也是为什么 Rust 强调「所有权」——堆数据的释放责任必须明确归属某个栈上的所有者，函数返回时随所有者一起 drop。
